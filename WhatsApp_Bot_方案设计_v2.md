## 一、业务背景

**所属业务线**：欢乐童年（VIPTHINK）港澳/海外 LP 团队

|---------|------|---------|

## 三、第一轮攻防：主动触达策略（5月30日 20:24）

### 3.1 🔴 红方（激进派）主张

**发言人 A**：

当前方案的主动触达只有 P2-1「沉默唤醒」一条，且定义为「7天未互动」，这个设计有几个致命问题：

| 问题 | 具体表现 |
|------|---------|
| **窗口太窄** | 7天在海外教育场景太短——港澳家长工作忙，1-2周不回消息是常态 |
| **触发条件单一** | 只盯「是否互动」，没盯「续费窗口期」「课时消耗」「退费风险」 |
| **没有分层** | 高意向家长和沉默家长用同一套话术？浪费机会 |
| **缺少节奏感** | 主动触达应该是战役式的（campaign），不是单点触发 |

**提出的5条策略线：**

```
策略线① 续费倒计时触达
├─ 触发条件：距下次续费日 ≤ 30天，且当前续费状态 = 未付
├─ 触达节奏：30天→15天→7天→3天→当日，共5次
├─ 每次话术递进：提醒→价值强化→限时优惠→紧迫感→最后通牒
└─ 数据联动：从SmartBI续费报表拉取到期日

策略线② 课时消耗预警触达
├─ 触发条件：剩余课时 ≤ 5节，且近7天未约课
├─ 触达动作：提醒剩余课时+推荐续费套餐+一键约课链接
└─ 场景价值：不等沉默再唤醒，在「即将用完」时主动介入

策略线③ 里程碑触达
├─ 学满30节/60节/100节 → 祝贺+成长报告+续费引导
├─ 成绩进步/竞赛获奖 → 个性化祝贺+转介绍引导
└─ 场景价值：正向激励比焦虑营销转化率高30%+

策略线④ 事件驱动触达
├─ 新课上线/假期特训班/呈分试冲刺 → 主动推送
├─ 港澳呈分试/DSE考前 → 针对性课程推荐
└─ 场景价值：借外部事件创造沟通理由

策略线⑤ 流失预警触达
├─ 触发条件：连续14天未约课 + 未回复消息
├─ 动作：LP主管亲自介入 → 专属优惠 → 停课挽留方案
└─ 场景价值：最后一道防线，挽回率约15-20%
```

### 3.2 🔵 蓝方（保守派）反驳

**发言人 B**：

红方方案看起来很完整，但忽略了一个核心风险：**WhatsApp 是私人通讯工具，不是营销渠道**。

| 风险 | 后果 |
|------|------|
| **模板审核风险** | 5条策略线每条需要独立模板，Meta审核周期2-5个工作日，改了话术要重新审 |
| **用户投诉风险** | WhatsApp对营销消息容忍度极低，投诉两次就封号——一整个业务线的号码被封，后果谁担？ |
| **24h窗口限制** | 主动触达必须用模板消息，模板消息必须通过审核，且每次都要付费（$0.005-0.09/条） |
| **用户体验反噬** | 港澳家长对推销敏感，如果30天内收到5条续费消息+3条课时提醒+2条活动推送，大概率拉黑 |

**提出的「三不原则」：**

```
① 不推送纯营销内容 —— 每条消息必须有服务价值
② 不超过频次上限 —— 每月主动触达 ≤ 2条/人
③ 不替代LP人工 —— Bot仅做预热和筛选，关键转化仍由LP完成

修正后的策略：合并为「1+2模型」
├─ 1条固定：续费窗口期提醒（仅限到期前15天和7天，共2次）
├─ 2条浮动：课时预警 / 里程碑祝贺（二选一，取优先级高者）
└─ 所有模板消息末尾带「回复1了解更多」引导用户进入24h窗口
```

### 3.3 ⚖️ 第一轮结论：3+1 漏斗触达模型

**合成方案**：红方覆盖度 + 蓝方安全边界 = 精准分层触达

```
T1 续费窗口触达 — 到期前15天+7天，月≤2条，不可跳过
T2 预警触发 — 课时<5节 · 14天沉默 · 未约课，条件触发
T3 事件驱动 — 新课·假期·呈分试·竞赛，人工审批后发送

安全兜底机制：
├─ 频控：月总≤3条/人, 周≤1条
├─ 内容审核：每条末端带回复引导
├─ 投诉熔断：单次投诉自动暂停7天
├─ 灰度发布：先跑10%人群1周看投诉率
├─ LP可见：所有自动消息同步推送LP知晓
├─ 退订机制：每消息含"回复停"退订
└─ 港版合规：遵守《个人资料(私隐)条例》

核心原则：Bot 做筛选和预热，LP 做转化和关单
```

## 四、第二轮攻防：续费率最大化的业务方案优化（5月30日 20:34）

### 4.1 🔴 红方（激进派）主张

**发言人 A**：

当前方案四层架构（触达→API→AI引擎→数据），**技术骨架是完整的，但缺少一个关键层——「转化策略层」**。从续费率最大化角度，有4个被忽略的杠杆：

**杠杆① 家长分层 × 差异化策略**

```
┌────────────────┬──────────────────────┬──────────────────┐
│ 家长类型        │ 占比（估计）          │ 续费策略          │
├────────────────┼──────────────────────┼──────────────────┤
│ A-主动续费型    │ ~30%                 │ 被动等待，不要干扰 │
│ B-犹豫观望型    │ ~40%                 │ 价值强化+限时优惠  │
│ C-价格敏感型    │ ~20%                 │ 分拆方案+分期支付  │
│ D-流失倾向型    │ ~10%                 │ 专属挽留组合拳     │
└────────────────┴──────────────────────┴──────────────────┘
```

**杠杆② 时机优化 × 行为触发**

续费不是「到期前才做的事」，而是从「首课结束」就开始：

```
首课结束 → 7天 → 30天 → 60天 → 90天(到期)
  │         │       │       │       │
  满意度     学习     进步     续费     紧急
  调研→     报告→    里程碑→  窗口→    挽留→
```

**杠杆③ 社交裂变 × 转介绍杠杆**

续费意愿最高的家长 = 已完成转介绍的家长（NPS净推荐值效应）：
→ 转介绍成功 → 自动发优惠券 → 续费时抵扣
→ "介绍一位朋友，双方各减200港币"

**杠杆④ 动态定价 × 紧迫感机制**

→ 早鸟价：到期前30天续费享9折
→ 阶梯价：续费课时包越大单价越低
→ 限时窗口：48小时专属优惠

### 4.2 🔵 蓝方（保守派）反驳

**发言人 B**：

红方的四个杠杆放到港澳教育市场会水土不服：

| 红方方案 | 港澳市场现实 |
|----------|-------------|
| 家长分层自动匹配策略 | 港澳家长反感「被标签」，且SmartBI录音转写大部分为空，分层准确率存疑 |
| 全程节点触发 | 港澳家长平均 WhatsApp 未读数 99+，学习报告大概率淹没在群消息里 |
| 转介绍裂变 | 港澳中产家长对「社交裂变」天然抵触，发优惠券 = 自降品牌格调 |
| 动态定价 | VIPTHINK 的定价权在总部，LP 团队无权动态调价——这是组织架构问题，不是技术问题 |

**提出的修正方案：从「推」转向「拉」**

```
① 服务质量 → 续费意愿
   ├─ 首课服务满意度 ≥ 4.5/5 → 续费率 +35%
   ├─ 家长问题响应时间 < 30min → 满意度 +20%
   └─ WhatsApp Bot 的核心价值：7×24响应，而非7×24推销

② 学习效果可视化 → 续费决策
   ├─ 每月自动生成「学员成长报告」→ WhatsApp 发送
   ├─ 含：课时完成率、知识点掌握度、同年龄段排名
   └─ 数据来自 SmartBI 学员画像表，不依赖录音转写

③ 降低决策门槛 → 提高转化
   ├─ Bot 自动推送「一键续费链接」+ 支付方式引导
   ├─ 港澳家长偏好：转数快(FPS) > 支付宝 > 信用卡
   └─ 路径越短，流失越少

④ 信任建立 → 长期留存
   ├─ 新生：3次人工电话（首课/7天/30天）+ Bot 补位
   ├─ 老生：Bot 日常维护 + 关键节点 LP 出面
   └─ 信任是续费的基础，不是优惠券
```

### 4.3 ⚖️ 第二轮结论：四支柱优化模型

**合成方案**：取红方之精准 纳蓝方之稳健 = 服务驱动型续费引擎

```
支柱一：服务质量护城河（权重35%）
├─ Bot 7×24 秒级响应家长咨询（覆盖睡眠时段）
├─ 首课服务满意度自动调研 → 低分自动创建 LP工单
├─ 家长提问未解决 → 15min 内转人工 + 钉钉告警
└─ 问题解决率纳入 LP 质检日报

支柱二：学习效果可视化（权重25%）
├─ 月维度自动生成「学员成长报告」WhatsApp推送
├─ 含课时完成率、知识点掌握度、同龄对比
├─ 里程碑节点：学满30/60/100节 → 祝贺+回顾
└─ 数据源：SmartBI 学员画像（不依赖录音转写）

支柱三：精准时机 × 行为触发（权重25%）
├─ 家长主动问「还有几节课」→ Bot识别 + 自动报价
├─ 学员连续7天高频约课 → 兴趣高峰 + 推荐升舱
├─ 续费倒计时30/15/7天节点自动触发
└─ 行为数据来源：SmartBI 约课/消耗/互动综合判断

支柱四：降低决策门槛（权重15%）
├─ 一键续费链接 + FPS/支付宝/信用卡多选
├─ 新生→老生过渡：3次人工+Bot补位（信任建立）
├─ 已转介绍家长 → 续费自动抵扣转介绍优惠
└─ 退费意向识别 → Bot安抚 + 即时转LP挽留
```

**续费率公式重构**：

```
旧模型：续费率 = 到期人数中付费人数占比（被动统计）
新模型：续费率 = 服务质量(35%) + 效果可见(25%) + 精准时机(25%) + 决策门槛(15%)
```

**架构变化**：在原四层架构上新增「转化策略层」作为第 2.5 层（L3）

## 五、个性化策略：从分层到一人一策（5月30日 20:46）

### 5.1 问题提出

**孙昊天**：分层策略是针对家长做了分层后的群体策略。有什么办法做个性化策略，因为LP能做到对每个家长的个性化续费。

### 5.2 核心思路

LP能个性化的原因 → 拆解为数据可采集的维度：

| LP的心理活动 | 对应数据维度 | 数据可得性 |
|-------------|-------------|-----------|

| 「上个月聊过价格问题，这次要给优惠方案」 | 交互历史摘要 | WhatsApp 对话日志 ✅ |
| 「陈太喜欢粤语沟通，不要太正式」 | 语言偏好 + 沟通风格 | 对话日志分析 ✅ |
| 「她是转介绍来的，多提一下介绍人」 | 转介绍关系链 | 续费系统 ✅ |
| 「最近几次课都没约，可能想停了」 | 约课频率趋势 | 约课系统 ✅ |
| 「上次说跟爸爸商量，这次该问结果了」 | 跟进状态机 | 对话日志 + CRM ✅ |

**结论：LP能做的个性化，90%可以用现有数据还原。**

### 5.5 个性化生成引擎

关键不是写死模板，而是把档案作为 **System Prompt 注入**：

```
System: 你是 VIPTHINK 港澳 LP 团队的续费助手。
当前沟通对象档案：
- 称呼：陈太
- 孩子：小明，P6，数学薄弱，正在准备呈分试
- 沟通风格：粤语，亲切但不啰嗦
- 上次沟通：5月20日聊过续费价格，陈太说"同先生商量下"
- 关键顾虑：觉得价格偏高，但认可教学效果
- 当前状态：课程6月15日到期，还剩8节课

任务：生成一条WhatsApp续费提醒消息
策略：先肯定进步 → 提呈分试紧迫性 → 给早鸟优惠 → 轻推决定
```

引擎三合一：**策略选择器 + 话术注入器 + 语调适配器**。

同一个「续费提醒」场景，三个不同家长收到完全不同的消息：

| 家长 | 策略 | 示例消息 |
|------|------|---------|
| 陈太 · P6呈分试 · 价格敏感 | 效果背书+价格换算+紧迫感 | 「陈太上堂數學最近進步好大喎，呈分試前呢個階段最關鍵，依家續費有早鳥優惠，每日唔使$50，呢個價真係好抵㗎！」 |
| Mr. Lee · 国际学校 · 效果驱动 | 数据佐证+进步量化+咨询式 | "Kevin has completed 45 sessions with 92% homework accuracy. His writing improved from Level 3 to 5. Shall we discuss his next phase plan?" |
| 张妈 · 沉默型 · 已2周未回 | 正面开场+情感钩子+低门槛邀请 | 「张妈好，小明最近两次课状态特别好，老师专门表扬了。他的课程快到期了，想跟您约个5分钟电话聊聊后续规划？」 |

### 5.7 与分层策略的关系

分层策略和个性化策略不是二选一，而是**两层叠加**：

| 层级 | 作用 | 举例 |
|------|------|------|
| **L1 分层** | 决定「是否触达 + 触达频率」 | B型家长月2次，D型家长紧急升级 |
| **L2 个性化** | 决定「说什么 + 怎么说」 | 同一句话，给陈太用粤语讲价格，给 Mr. Lee 用英语讲数据 |

### 5.8 LP影子模式（渐进替代）

```
阶段1：LP 正常发消息 → Bot 静默记录（学）
阶段2：Bot 生成消息 → LP 审核后发送（半自动）
阶段3：Bot 生成 + 自动发送 → LP 事后抽查（大半自动）
阶段4：Bot 全自动 → LP 只看异常（全自动）

每个阶段根据家长的"信任分"独立晋级：
- 信任分 > 0.8 → 进入阶段4
- 信任分 0.5-0.8 → 阶段3
- 信任分 < 0.5 → 阶段2
- 关键场景（退费/投诉）→ 始终阶段1
```

## 六、第三轮攻防：该推谁，规则引擎还是个性化引擎？（5月30日 21:03）

### 6.1 问题提出

**孙昊天**：为什么要让分层策略管「该推谁」，为什么还让它用规则引擎。个性化策略为什么不能管该推谁？

### 6.2 🔴 红方（统一引擎派）主张

**发言人 A**：

之前给出的「分层管该推谁 + 个性化管说什么」这个二分法，本质上是一个**工业时代的遗留思维**——流水线思维：第一道工序筛人，第二道工序加工话术。

但 **LP 本人「该推谁」和「推什么」是同一个决策过程，不是两道工序。**

把这两者拆开，会制造三类致命断层：

| 断层类型 | 具体场景 | 后果 |
|---------|---------|------|
| **边界模糊断层** | 规则说「B型家长到期前15天推」，但张妈虽属B型，昨天刚在群里抱怨老师——推还是不推？规则不知道，规则只管推。 | 在家长情绪最差的时候推续费 = 火上浇油 |
| **时机错失断层** | 规则说「沉默14天才唤醒」，但李生虽然上周有互动，那次互动是「我考虑一下」——其实比沉默14天的家长更需要推 | 规则永远追不上真实信号 |
| **策略协同断层** | 规则标记「该推」，个性化引擎生成了一封热情洋溢的续费消息——但规则不知道上次 LP 承诺的是「不催你，等你想好」 | 前后矛盾，信任崩塌 |

**主张：废除「该推谁」的独立规则层，把决策权还给个性化引擎。**

```
新流程：
家长数字孪生档案（五维数据）
        ↓
  LLM 综合评估：
  ├─ 是否需要触达？（基于全部上下文判断）
  ├─ 什么时候触达？（基于家长的回复窗口）
  ├─ 触达什么内容？（基于家长画像+历史）
  ├─ 用什么语气？（基于沟通风格）
  └─ 走什么渠道？（基于渠道偏好）
        ↓
  一条消息输出（而不是两步流水线）
```

### 6.3 🔵 蓝方（分层守护派）反驳

**发言人 B**：

红方说的「LP是整体决策」完全同意。但**这个系统的操作者不是孙昊天一个人，而是一个团队，面对的是 WhatsApp 平台和港澳法律。**

纯 LLM 管「该推谁」有三个不可接受的风险：

| 风险 | 具体场景 | 后果严重性 |
|------|---------|-----------|
| **合规风险** | LLM 判断某沉默家长「值得推」，自动发了一条营销消息。但 WhatsApp 规定：24h窗口外的主动触达必须用**已审核模板**。LLM 不知道这个规则，或者知道但「创造性」地绕过了 | **封号。整个业务线断联。不可逆。** |
| **一致性风险** | 同一个家长，周一 LLM 判断不该推，周三换了个 LLM 版本判断该推——LP 团队无法预测系统行为，无法向家长解释 | 运营失控 |
| **审计风险** | 某家长投诉「你们骚扰我」，需要回溯：为什么系统当时决定推这条消息？LLM 的决策链不可追溯——「模型当时觉得合适」在法律上不是有效解释 | 法律纠纷 |

**规则引擎不是落后的工业思维，而是安全气囊。它的价值不是「聪明」，而是「确定」。**

```
正确架构：
┌─────────────────────────────────────┐
│  安全边界层（规则引擎·硬约束）        │
│  ├─ 24h外非模板不发（WhatsApp合规）   │
│  ├─ 月触达>3条不发（频控）            │
│  ├─ 投诉冷却期内不发（熔断）          │
│  ├─ 退费中/纠纷中不发（红线）         │
│  └─ 港澳私隐条例合规检查              │
└──────────────┬──────────────────────┘
               ↓ 通过安全检查后
┌─────────────────────────────────────┐
│  智能决策层（LLM·软建议）             │
│  ├─ 现在是不是好时机？（综合判断）      │
│  ├─ 该用什么策略？（画像匹配）          │
│  ├─ 该说什么？（个性化生成）           │
│  └─ 建议优先级（高/中/低）             │
└─────────────────────────────────────┘
```

**规则管「绝对不发」，LLM 管「发什么最合适」。边界清晰，各司其职。**

### 6.4 红方反攻

**发言人 A**：

蓝方说的三个风险，逐一反驳：

**① 「WhatsApp 24h 规则 LLM 不知道」→ 谁说不知道？**

把规则写进 System Prompt 就行了：
```
System: 你必须严格遵守以下硬性规则，不可绕过：
1. 家长上次发消息距今 > 24小时 → 只能用审核通过的模板消息
2. 该家长本月已收 ≥ 3条消息 → 不发送
3. 该家长在投诉冷却期 → 不发送
...
如违反上述任何一条，回复 "DO_NOT_SEND"
```

这不叫「规则引擎」，这叫「规则注入到 LLM 上下文中」。LLM 遵循指令的能力已被充分验证。

**② 「一致性风险」→ 规则引擎一样有**

规则引擎的规则也会被人改。谁改的？什么时候改的？为什么改？同样需要审计。

**③ 「审计风险」→ 可解决**

LLM 每次决策时记录：
```
决策日志：
├─ 输入：家长档案快照（可追溯）
├─ System Prompt 版本（可追溯）
├─ LLM 推理链
├─ 最终决策：SEND / DO_NOT_SEND
└─ 生成内容
```

这个审计链比规则引擎更详细——规则引擎只会说「触发了规则3」，不会解释「为什么规则3适用」。

### 6.5 蓝方反攻

**发言人 B**：

红方把「规则注入 Prompt」等同于规则引擎，这偷换了概念。

**核心区别：Prompt 中的规则是「建议」，代码中的规则是「执行」。**

| 场景 | Prompt规则（红方方案） | 代码规则（蓝方方案） |
|------|----------------------|---------------------|
| 某家长在投诉冷却期 | LLM 判断「应该不发」 | 代码直接阻断 |
| LLM 出现幻觉 | **绕过规则，照发不误** | 代码不可绕过 |
| 凌晨3点批量推送 | LLM 可能误判 | 代码精确控制 |
| 上级紧急要求暂停所有推送 | 需要等 LLM 下一次推理才知道停了没 | 代码即时生效 |

**你的 System Prompt 写得再好，LLM 也是一个概率系统——不是确定性系统。**「大概率不发」和「绝对不发」在 WhatsApp 封号这件事上，差距是一条业务线的生死。

**成本问题**：
- 规则引擎：5000个家长 → 规则跑一遍 → 筛出200个该推的 → LLM 只为这200个生成消息
- 纯 LLM：5000个家长 → LLM 逐个判断 → 5000次推理 → 成本是前者的25倍

### 6.6 ⚖️ 第三轮结论：规则包围 LLM

**合成方案**：红方赢了原则，蓝方赢了设计。

```
阶段一：规则粗筛（低成本确定性）
→ 代码规则瞬间筛掉绝对不发的人群：
  24h外无模板 / 月超频 / 投诉冷却 / 退费中 / 已续费
→ 通过 → 进入候选池

阶段二：LLM 智能排序（高精度决策）
→ 输入：候选池中每个家长的完整五维档案
→ 任务：评估触达优先级（1-10分）+ 推荐策略 + 推荐时机
→ 输出：排序后的触达列表（附每条的消息草稿）
→ 关键：LLM 看到的不是「该不该推」的二元判断，而是完整的「五维档案 + 历史上下文」→ 整体决策

阶段三：频控阀（规则代码兜底）
→ 按优先级逐个消耗：月度总预算200条 → 排前面的先发 → 预算耗尽自动停
→ 每条发送前再做一次规则检查（防LLM幻觉绕过）→ 真正的双保险
→ 最终发送
```

**关键设计：规则包围 LLM，而非规则前置于 LLM**

| | 错误做法 | 正确做法 |
|------|---------|---------|
| 流程 | 规则先筛→LLM只负责话术（割裂、丢失上下文） | 规则粗筛去噪声→LLM拿着完整上下文做整体决策→规则再兜底（双层安全） |
| 规则角色 | 前置过滤器（割裂决策） | 包围式安全罩（粗筛 + 兜底） |
| LLM角色 | 只负责话术生成 | 同时决定策略+时机+话术（整体决策） |

## 七、最终方案：五层架构 v2

从最初的「四层」扩展为「五层」，新增 L3 转化策略层：

```
L1 家长触达层
├─ 港澳家长 / 欧美澳家长
├─ 被动咨询 / 主动触达
└─ 沉默唤醒（3+1漏斗）

L2 API 接入层
├─ 阿里云 Chat App（推荐）
├─ Webhook Worker
├─ 模板审核管理
└─ 24h智能切换

L3 转化策略层（NEW）
├─ 规则安全罩（代码）：24h外无模板不发 / 月超频不发 / 投诉冷却不发 / 预算控制
├─ LLM 决策芯（智能）：五维档案评估 → 优先级排序 → 策略时机推荐 → 消息草稿
└─ 四支柱优化：服务质量 / 效果可视化 / 精准时机 / 降低门槛

L4 AI 引擎层
├─ 意图识别：L1关键词→L2向量→L3 LLM（复用现有路由引擎）
├─ 知识库：LP规则+SOP+话术（复用现有knowledge库）
├─ 个性化引擎：策略·话术·语调三合一适配器
├─ LP影子模式：学习→建议→自动，信任分渐进替代
└─ 人工升级：敏感场景→LP，钉钉即时告警

L5 数据与运营层
├─ 数字孪生档案：五维JSON实时更新
├─ 对话日志：全量存+摘要索引
├─ 续费率看板：过程指标+结果指标
├─ 质检数据：WhatsApp纳入质检
└─ 反馈闭环
```

**与现有系统联动**：

| 现有系统 | 联动方式 |
|----------|---------|
| `vipthink-lp-assistant`（钉钉Bot） | 共用知识库、共用路由引擎，仅渠道不同 |
| `vipthink-renewal-intention`（续费意向） | WhatsApp 信号自动回写飞书表格 |
| SmartBI 质检日报 | WhatsApp 对话数据纳入质检数据源 |
| 飞书 Lark CLI | 用于推送异常告警和运营统计 |
| 钉钉 Webhook | 敏感场景转人工时推送通知 |

**核心原则**：
- Bot 做筛选和预热，LP 做转化和关单
- 规则包围 LLM，而非前置 LLM
